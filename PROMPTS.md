# Пошаговый сценарий промптов для Claude Code

Это твой сценарий разговора с Claude Code. Иди шагами, не прыгай через ступени, не объединяй фазы. После каждого шага — проверяй результат, и только потом давай следующий промпт.

Каждый промпт сформулирован так, чтобы Claude Code понимал контекст и не сваливался в инициативу.

---

## Шаг 0 — стартовый. Положи INSTRUCTIONS.md и MIGRATION_PLAN.md в корень

Перед запуском Claude Code:
- Скопируй файлы `INSTRUCTIONS.md` и `MIGRATION_PLAN.md` в **корень проекта** (туда же где `frontend/`).
- Структура должна выглядеть так:
  ```
  sfu_vector-master/
  ├── INSTRUCTIONS.md
  ├── MIGRATION_PLAN.md
  └── frontend/
      ├── maket/      (read-only исходник дизайна)
      └── web/        (рабочий Next-проект)
  ```
- Запусти Claude Code из папки `sfu_vector-master/`.

## Если что-то идёт не так

**Сценарий 1: Claude Code начал использовать Tailwind**
```
Стоп. В INSTRUCTIONS.md явно сказано — НИКАКОГО TAILWIND. У нас inline-styles и кастомные utility-классы. Откати последние изменения (git checkout или восстанови по памяти) и переделай этот файл, используя style={{}} и className только для существующих классов из globals.css.
```

**Сценарий 2: Claude Code предлагает поставить новую библиотеку**
```
Нет. В INSTRUCTIONS.md явно сказано — никаких новых библиотек без моего разрешения. Реализуй на нативных средствах React + inline-SVG. Если задача правда требует внешнюю либу — обоснуй и спроси меня.
```

**Сценарий 3: Claude Code решил сделать рефакторинг существующего компонента**
```
Стоп. Я не просил рефакторить существующие компоненты. Откати изменения в этом файле, верни как было, и продолжай миграцию по плану.
```

**Сценарий 4: Claude Code сломал существующую страницу**
```
Существующая страница [имя] сломалась после твоих изменений. Откати последние изменения в Sidebar (или другом файле) и попробуй другим способом — возможно через дефолтное значение пропа.
```

**Сценарий 5: типы ругаются и не компилируется**
```
Сначала разберись в чём проблема типов, потом исправляй. Не используй "any" без необходимости. Если структура данных в mock-data.ts не соответствует ожиданиям компонента — возможно компонент написан с расчётом на старую структуру, проверь maket-источник.
```

## Финал

После прохождения всех фаз у тебя должен быть полностью функциональный MVP с переходами между экранами, моками, и без пустых кнопок (где-то реальные навигации, где-то TODO-заглушки с консоль-логом).

Дальше уже:
- Подключение реального бэкенда (API).
- Реальная авторизация (NextAuth, Clerk, или своё).
- Реальная загрузка картинок (S3 / Cloudinary).
- QR-код (qrcode-svg или сервис).
- Карта (Яндекс / 2GIS / Mapbox).

Но это уже не дизайн-перенос, это разработка функционала.
