# Инструкции для Claude Code — миграция дизайна СФУ.Вектор в Next-проект

## Контекст проекта

Я веду университетскую платформу «СФУ.Вектор» — единое пространство студенческих возможностей в Сибирском федеральном университете. У меня **два каталога с кодом**:

- `frontend/web/` — рабочий Next.js-проект (Next 16 + React 19 + TypeScript), куда будем переносить дизайн.
- `frontend/maket/` — папка с **готовым дизайном** в виде JSX-файлов. Это «исходник дизайна», из него мы тащим разметку. ВНУТРИ MAKET НИЧЕГО НЕ МЕНЯЕМ — это reference, read-only.

Дизайн уже доведён до апрува (мы делали его пошагово в claude.ai/design с ревью). Сейчас задача — **аккуратно перенести его в Next**, превратив JSX в TSX, соединить страницы между собой через `next/navigation`, и сделать состояние/моки чтобы интерактивные элементы работали.

## Что менять и что НЕ менять

### НЕ менять (это закон)

1. **Никакого Tailwind.** В проекте нет Tailwind и не должно появиться. Стили только через `style={{}}` и `className` для уже существующих utility-классов из `app/globals.css` (.btn, .btn-primary, .btn-ghost, .card, .chip, .input, .h1-.h4, и т.д.).
2. **Никаких UI-китов.** Не shadcn/ui, не MUI, не Chakra, не Radix, не Headless UI. Всё на нативных HTML-элементах + inline-SVG.
3. **Никаких новых библиотек** без явного разрешения. Если кажется что нужна — спроси. Особенно: не добавлять lucide-react, react-icons, date-fns, dayjs, react-hook-form, zod, framer-motion. Если что-то очень нужно — обоснуй и спроси.
4. **Не менять CSS-переменные** в `globals.css` (--bg, --fg, --blue, --grad, и т.д.). Они уже подобраны под бренд «Бумага».
5. **Не менять имена существующих компонентов** в `components/` (Sidebar, Topbar, Footer, EventCover, NotificationsBell, Wheel, DormsMap). Их можно расширять (добавлять пропы), но не переименовывать и не дробить без необходимости.
6. **Не переписывать landing-секции** в `components/landing/` — они уже сделаны и работают.

### Менять (по плану из MIGRATION_PLAN.md)

1. Создавать новые страницы под Next App Router в `app/dashboard/...` и `app/org/[slug]/...`.
2. Создавать новые компоненты в `components/` по тематическим папкам.
3. Расширять `lib/mock-data.ts` — добавлять новые сущности (Organization, Application, Registration, и т.д.) согласно MIGRATION_PLAN.md.
4. Менять `components/Sidebar.tsx` — добавить ContextSwitcher (см. дизайн).
5. Делать кнопки и ссылки кликабельными через `next/link` и `useRouter` из `next/navigation`.

## Принципы переноса JSX → TSX

### 1. Из MAKET читать, в WEB писать

Дизайн лежит в `frontend/maket/screens/*.jsx`. Это **исходник**. Никогда не пиши в maket. Только читай.

Пишешь в `frontend/web/`. Если по ошибке потянет руку отредактировать что-то в maket — остановись и спроси.

### 2. JSX → TSX: что меняется

- Расширение файла `.jsx` → `.tsx`.
- Все компоненты типизируются. Если у функции есть пропы — описывай интерфейс пропов сверху файла.
- `window.SomeComponent = SomeComponent` в конце файла **УДАЛЯЕМ** — это было нужно только для дизайн-канваса с babel-standalone. В Next используем `export` или `export default`.
- Импорты других компонентов — через `@/components/...` (alias уже настроен в tsconfig).
- Импорты данных из `@/lib/mock-data`.
- Иконочные модули (Ic, Sb, Db, Ap, Cw, Rg в дизайне) — каждый локально, в начале своего файла. Не пытайся унифицировать их сейчас.

### 3. Кнопки и ссылки

Это критично. В дизайне почти все кнопки — это `<button>` без обработчика. В Next нужно:

- Если кнопка ведёт на другую страницу — оборачивай в `<Link href="...">` или используй `useRouter().push("...")`. Используй ссылки и роутер из `'next/navigation'`, НЕ из `'next/router'` (тот legacy).
- Если кнопка переключает состояние внутри страницы — `useState` в той же странице с `'use client'`.
- Если назначение кнопки неочевидно — поставь `onClick={() => console.log('TODO: ...')}` и оставь `// TODO` коммент. Не оставляй пустых обработчиков молча.
- См. MIGRATION_PLAN.md — там для каждого экрана таблица «кнопка → куда ведёт».

### 4. Серверные vs клиентские компоненты

- Если компонент использует `useState`, `useEffect`, `useRouter`, обработчики событий — на самом верху файла должно стоять `'use client'`.
- Если просто разметка без интерактива — оставляем серверным (без 'use client').
- В сомнениях — ставь `'use client'`. Не оптимизируй преждевременно.

### 5. Структура App Router

App Router использует папки = роуты. Один экран = одна папка с `page.tsx`. Для динамических роутов — `[id]`, `[slug]`.

Layouts (`layout.tsx`) — общий каркас для всех страниц внутри. У нас уже есть `app/dashboard/layout.tsx` с Sidebar + Topbar + Footer. Новые dashboard-страницы используют его автоматически.

Для org-режима нужен **отдельный layout** `app/org/[slug]/layout.tsx`, который рендерит Sidebar в org-режиме.

## Стиль кода

- Двойные кавычки в JSX-атрибутах, одинарные в JS-строках (как в существующих компонентах).
- Пробелы 2 знака.
- Никаких комментариев типа `// imports` или `// state` — читается и так.
- Не оборачивать всё в `useMemo`/`useCallback` без причины — это дизайн-перенос, преждевременная оптимизация только мешает.
- Стейт мокаем хардкодом: пользователь = Иван Петров (см. CURRENT_USER в mock-data), он же залогинен везде.

## Если что-то не понятно

**Спроси.** Это лучше чем сделать неправильно и пройти 5 экранов с одним и тем же багом. Особенно:

- Если кажется что нужна новая библиотека.
- Если кажется что нужно поменять CSS-переменную.
- Если кнопка в дизайне неочевидно куда ведёт.
- Если дизайн противоречит уже существующему коду в проекте.
